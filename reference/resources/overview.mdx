## What are Resources?

A Resource is a network-accessible external service that your application depends on. Resources are defined in `convox.yml` and linked to one or more [Services](/reference/primitives/app/service), making them available through environment variables.

## Basic Usage

Define resources in your `convox.yml` and link them to services:

```yaml
resources:
  database:
    type: postgres
  cache:
    type: redis

services:
  web:
    resources:
      - database
      - cache
```

## Environment Variables

When a resource is linked to a service, Convox automatically injects environment variables with connection details:

<ParamField path="RESOURCE_URL" type="string">
  Complete connection URL for the resource
</ParamField>

<ParamField path="RESOURCE_HOST" type="string">
  Hostname of the resource
</ParamField>

<ParamField path="RESOURCE_PORT" type="string">
  Port number of the resource
</ParamField>

<ParamField path="RESOURCE_USER" type="string">
  Username for authentication (databases only)
</ParamField>

<ParamField path="RESOURCE_PASS" type="string">
  Password for authentication (databases only)
</ParamField>

<ParamField path="RESOURCE_NAME" type="string">
  Database name (databases only)
</ParamField>

### Example

For a `postgres` resource named `database`:

```bash
DATABASE_URL=postgres://username:password@host.name:5432/database
DATABASE_USER=username
DATABASE_PASS=password
DATABASE_HOST=host.name
DATABASE_PORT=5432
DATABASE_NAME=database
```

## Containerized vs Managed Resources

### Containerized Resources

By default, resources run as containers within your Rack. This approach:

- Gets you up and running quickly
- Provides cost-effective solutions for development/staging
- Runs alongside your application services

Supported types: `postgres`, `mysql`, `mariadb`, `redis`, `memcached`, `postgis`

### Managed Resources

For production environments, use managed cloud services:

- **AWS RDS**: `rds-postgres`, `rds-mysql`, `rds-mariadb`
- **AWS ElastiCache**: `elasticache-redis`, `elasticache-memcached`

<Note>
Managed resources provide enhanced durability, automated backups, high availability, and operational best practices.
</Note>

## Custom Images

You can use custom Docker images for containerized resources:

```yaml
resources:
  database:
    type: postgres
    image: pgvector/pgvector:pg16
```

<Warning>
Always include an image tag. If omitted, `:latest` will be used, which may cause unexpected behavior.
</Warning>

## Resource Overlays

Resource overlays allow you to use containerized resources in some environments and managed services in others.

To overlay a resource, set its environment variable manually:

```bash
convox env set DATABASE_URL=postgres://user:pass@prod-db.amazonaws.com:5432/db -a myapp -r production
```

When the `DATABASE_URL` environment variable is set, Convox won't start the containerized resource. Your app will use the external database instead.

<Note>
This pattern is ideal for using local resources in development and managed services in production.
</Note>

## CLI Commands

### List Resources

```bash
convox resources -a myapp
```

### Get Resource Info

```bash
convox resources info <name> -a myapp
```

### Get Resource URL

```bash
convox resources url <name> -a myapp
```

### Start a Console

```bash
convox resources console <name> -a myapp
```

### Proxy to Resource

```bash
convox resources proxy <name> -a myapp
```

This creates a local tunnel, allowing you to connect local tools to resources running in the Rack.

### Export Data

```bash
convox resources export <name> -f /tmp/backup.sql -a myapp
```

### Import Data

```bash
convox resources import <name> -f /tmp/backup.sql -a myapp
```

<Note>
Export and import are only available for database resources (postgres, mysql, mariadb).
</Note>
