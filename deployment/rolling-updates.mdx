When a [Release](/reference/primitives/app/release) is promoted, Convox performs a graceful rolling update to avoid disruption to your [App](/reference/primitives/app). New [Processes](/reference/primitives/app/process) are incrementally deployed while maintaining service uptime and capacity.

## How Rolling Updates Work

Convox uses a "make one, break one" process to maintain [Service](/reference/primitives/app/service) uptime and capacity during deployments.

<Steps>
  <Step title="Start new Process">
    One new [Process](/reference/primitives/app/process) is started on the new [Release](/reference/primitives/app/release).
  </Step>
  
  <Step title="Health check">
    Convox verifies that the new Process passes a [health check](/configuration/health-checks) before proceeding.
  </Step>
  
  <Step title="Stop old Process">
    Once the new Process is healthy, one old Process running the previous Release is gracefully stopped.
  </Step>
  
  <Step title="Repeat">
    Steps 1-3 repeat until all Processes are running the new Release.
  </Step>
</Steps>

<Note>
This approach ensures that your application always has healthy processes available to handle requests throughout the deployment.
</Note>

## Deployment Configuration

You can control rolling update behavior using deployment configuration in your `convox.yml`:

```yaml
services:
  web:
    build: .
    port: 3000
    deployment:
      minimum: 50
      maximum: 200
```

### Configuration Options

| Attribute | Default | Description |
|-----------|---------|-------------|
| **minimum** | 50 | Minimum percentage of healthy processes that must be running during update |
| **maximum** | 200 | Maximum percentage of processes that can be running during update |

<Tip>
The default configuration maintains at least 50% capacity and allows up to 200% capacity during updates, ensuring smooth deployments.
</Tip>

## Understanding Min/Max Configuration

The minimum and maximum settings control how aggressively Convox can deploy your updates:

### Conservative Deployment (High Minimum)

```yaml
deployment:
  minimum: 75
  maximum: 125
```

**Characteristics**:
- Maintains high capacity throughout deployment
- Slower deployment as fewer processes change at once
- Best for high-traffic applications that can't tolerate capacity reduction

### Aggressive Deployment (Low Minimum)

```yaml
deployment:
  minimum: 25
  maximum: 200
```

**Characteristics**:
- Faster deployment with multiple parallel updates
- Lower minimum capacity during deployment
- Best for development or low-traffic environments

### Balanced Deployment (Default)

```yaml
deployment:
  minimum: 50
  maximum: 200
```

**Characteristics**:
- Good balance between speed and safety
- Suitable for most production workloads
- Recommended starting point

## Health Checks During Rolling Updates

Health checks are critical to the rolling update process. They determine when a new Process is ready to receive traffic.

### Basic Health Check

```yaml
services:
  web:
    health: /health
    port: 3000
```

### Advanced Health Check

```yaml
services:
  web:
    health:
      grace: 10
      interval: 5
      path: /health
      timeout: 3
    port: 3000
```

<Warning>
If your health checks are too strict or your application takes too long to start, rolling updates may fail and trigger an automatic rollback.
</Warning>

### Health Check Requirements

For successful rolling updates, your health endpoint must:

1. Return HTTP status 200-399
2. Respond within the configured timeout (default 4 seconds)
3. Check critical dependencies (database connections, required services)
4. Complete before the grace period expires

## Automatic Rollback

Convox automatically rolls back deployments if issues are detected during the rolling update.

### Rollback Triggers

Automatic rollback occurs when:

<CardGroup cols={2}>
  <Card title="Process Fails to Start" icon="circle-xmark">
    A Process fails to start up and listen on the expected port within the grace period.
  </Card>
  
  <Card title="Health Check Failure" icon="heart-crack">
    A Process fails to pass health checks after multiple attempts.
  </Card>
  
  <Card title="Crash Loop" icon="rotate-exclamation">
    A Process repeatedly crashes or restarts unexpectedly.
  </Card>
  
  <Card title="Resource Constraints" icon="memory">
    A Process cannot be scheduled due to insufficient resources.
  </Card>
</CardGroup>

### Rollback Process

When a rollback is triggered:

<Steps>
  <Step title="Deployment halts">
    The rolling update stops immediately.
  </Step>
  
  <Step title="New processes terminated">
    All new Processes running the failed Release are stopped.
  </Step>
  
  <Step title="Old processes restored">
    The previous Release's Processes are restored to their original count.
  </Step>
  
  <Step title="Logs provided">
    Detailed logs are shown to help diagnose the issue.
  </Step>
</Steps>

## Monitoring Rolling Updates

### Real-Time Logs

During a deployment, watch logs in real-time:

```bash
$ convox deploy -a myapp
# Logs are automatically streamed during deployment
```

### Separate Terminal for Logs

In a separate terminal, monitor application logs:

```bash
$ convox logs -a myapp --follow
```

### Check Process Status

Monitor process status during deployment:

```bash
$ convox ps -a myapp
ID                    SERVICE  STATUS   RELEASE      STARTED       COMMAND

web-6d4e734ab-xyz89   web      running  RBCDEFGHIJ   2 minutes ago

```

## Advanced Scenarios

### Database Migrations

For deployments requiring database migrations:

<Steps>
  <Step title="Build without promoting">
    ```bash
    $ convox build -a myapp
    Release: RBCDEFGHIJ
    ```
  </Step>
  
  <Step title="Run migrations">
    ```bash
    $ convox run web rake db:migrate --release RBCDEFGHIJ -a myapp
    ```
  </Step>
  
  <Step title="Promote the release">
    ```bash
    $ convox releases promote RBCDEFGHIJ -a myapp
    ```
  </Step>
</Steps>

### Zero-Process Deployments

If you need to deploy a service that's currently scaled to zero:

```bash
$ convox scale web --count=1 -a myapp
$ convox deploy -a myapp
```

### Blue-Green Deployments

For additional safety, use separate apps for blue-green deployments:

```bash
# Deploy to blue environment
$ convox deploy -a myapp-blue

# Test thoroughly, then switch traffic
$ convox switch myapp-blue myapp-green
```

## Troubleshooting

### Deployment Fails Immediately

**Symptoms**: Deployment fails before any new processes start

**Possible causes**:
- Build errors
- Invalid convox.yml configuration
- Resource constraints

**Solutions**:
1. Review build logs carefully
2. Validate convox.yml syntax
3. Check rack capacity with `convox rack`

### Health Checks Never Pass

**Symptoms**: New processes start but never become healthy

**Possible causes**:
- Application not listening on correct port
- Health endpoint returning wrong status code
- Grace period too short
- Application dependencies unavailable

**Solutions**:
1. Check application logs: `convox logs -a myapp`
2. Verify port configuration matches application
3. Test health endpoint manually
4. Increase grace period in health check config

### Deployment Hangs

**Symptoms**: Deployment starts but makes no progress

**Possible causes**:
- Insufficient resources to start new processes
- Image pull failures
- Network connectivity issues

**Solutions**:
1. Check process status: `convox ps -a myapp`
2. Review Kubernetes events in logs
3. Verify image registry is accessible
4. Check rack resource availability

### Slow Deployments

**Symptoms**: Deployments take too long to complete

**Possible causes**:
- Conservative deployment configuration
- Long health check intervals
- Large number of processes
- Slow image pulls

**Solutions**:
1. Adjust deployment minimum/maximum settings
2. Optimize health check timing
3. Use image caching strategies
4. Consider regional image registries
