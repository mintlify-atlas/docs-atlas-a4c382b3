Convox Workflows enable you to automate continuous integration and continuous delivery processes. They integrate directly with GitHub and GitLab to build, test, and deploy your applications automatically in response to code changes.

## Overview

Convox offers two types of workflows:

<CardGroup cols={2}>
  <Card title="Review Workflows" icon="code-pull-request">
    Automatically build and test pull requests, with optional temporary review apps for testing changes before merging.
  </Card>
  
  <Card title="Deployment Workflows" icon="rocket">
    Automatically deploy your application to staging and production when code is merged to specific branches.
  </Card>
</CardGroup>

## Review Workflows

Review workflows test new versions of your application based on pull requests. When configured, Convox automatically builds your application for every pull request, optionally runs tests, and can create temporary review applications.

### Creating a Review Workflow

<Steps>
  <Step title="Navigate to Workflows">
    In the Convox Console, go to the **Workflows** tab and click **Create Workflow**.
  </Step>
  
  <Step title="Select Review Workflow">
    Choose **Review Workflow** as the workflow type.
  </Step>
  
  <Step title="Configure settings">
    Set up your review workflow with the following options:
    
    | Setting | Description |
    |---------|-------------|
    | **Repository** | The connected GitHub or GitLab repository that will trigger the workflow |
    | **Manifest** | The Convox manifest file to use (defaults to `convox.yml`) |
    | **Rack** | The Rack where review applications will be deployed |
    | **Run tests** | Enable to run the test command specified in your `convox.yml` |
    | **Deploy Demo** | Enable to create a temporary review application that will be deleted when the PR is merged |
    | **Before Promote** | Specify a service and command to run before promotion (e.g., database migrations) |
    | **After Promote** | Specify a service and command to run after promotion (e.g., notifications) |
    | **Demo Environment** | Define environment variables for review applications |
  </Step>
</Steps>

### Review Workflow Behavior

When a review workflow is triggered:

<Steps>
  <Step title="Build">
    Convox builds the application using the code from the pull request.
  </Step>
  
  <Step title="Test">
    If enabled, tests are run against the build.
  </Step>
  
  <Step title="Deploy">
    If enabled, a review application is created with a unique URL.
  </Step>
  
  <Step title="Report">
    The workflow status is reported back to GitHub/GitLab.
  </Step>
  
  <Step title="Cleanup">
    When the PR is merged, the review application is automatically deleted.
  </Step>
</Steps>

<Tip>
Review applications are perfect for QA teams to test features before they reach production, and for stakeholders to preview changes in a real environment.
</Tip>

## Deployment Workflows

Deployment workflows automate the process of deploying your application to staging and production environments when code is merged into specified branches.

### Creating a Deployment Workflow

<Steps>
  <Step title="Navigate to Workflows">
    In the Convox Console, go to the **Workflows** tab and click **Create Workflow**.
  </Step>
  
  <Step title="Select Deployment Workflow">
    Choose **Deployment Workflow** as the workflow type.
  </Step>
  
  <Step title="Configure settings">
    Set up your deployment workflow:
    
    | Setting | Description |
    |---------|-------------|
    | **Repository** | The connected GitHub or GitLab repository |
    | **Workflow Name** | A descriptive name (e.g., "Staging Deploy") |
    | **Branch** | The branch that triggers the workflow when code is merged |
    | **Manifest** | The Convox manifest file to use (defaults to `convox.yml`) |
    | **Applications** | One or more applications to deploy to, with promotion settings |
    | **Run tests** | Enable to run the test command specified in your `convox.yml` |
    | **Before Promote** | Specify a service and command to run before promotion |
    | **After Promote** | Specify a service and command to run after promotion |
    | **Environment** | Define environment variables for the deployment |
  </Step>
  
  <Step title="Configure promotion">
    For each application, choose:
    
    - **Automatic**: The new build is automatically promoted after creation
    - **Manual**: The build is created but must be manually promoted
  </Step>
</Steps>

<Note>
Manual promotion is recommended for production environments to give you a final opportunity to review before deployment.
</Note>

### Manually Running a Workflow

You can manually trigger any deployment workflow:

<Steps>
  <Step title="Navigate to the workflow">
    Go to the **Workflows** tab and find your workflow.
  </Step>
  
  <Step title="Trigger the workflow">
    Click the **Play** button next to the workflow and confirm.
  </Step>
</Steps>

## Workflow Jobs

Each workflow run creates a job that can be monitored through the Convox Console.

### Viewing Jobs

<Steps>
  <Step title="Navigate to Jobs">
    Go to the **Jobs** tab in the Convox Console.
  </Step>
  
  <Step title="View job details">
    Click on any job to see:
    
    - Step-by-step execution logs
    - Build and release information
    - Deployment URLs for review applications
    - Error details if the workflow failed
    - Options to re-run the job
  </Step>
</Steps>

## CLI Management

Manage workflows via the Convox CLI for programmatic access and automation.

### List Workflows

```bash
$ convox workflows
ID                                    KIND        NAME
55dd9440-eb98-4d9b-816f-9923ee77feff  deployment  deploy-web-app
c828b45a-070b-46ed-9c43-ddaa905ecd68  review      review-web-app
```

### Trigger a Workflow

```bash
$ convox workflows run 55dd9440-eb98-4d9b-816f-9923ee77feff --branch feat-branch --title "Feature Branch Deploy"
Successfully triggered the workflow, job id: 65a4160a-27cd-47c6-ba74-aaaaaaaa
```

## Common Workflow Patterns

### Pattern 1: Pull Request Testing

Automatically deploy a temporary version for each pull request:

<Steps>
  <Step title="Create review workflow">
    Set up a review workflow for your repository.
  </Step>
  
  <Step title="Enable testing">
    Turn on "Run tests" and "Deploy demo".
  </Step>
  
  <Step title="Configure environment">
    Add any necessary environment variables for review apps.
  </Step>
</Steps>

**Result**: Every PR gets a fully functional preview URL for testing.

### Pattern 2: GitFlow Deployment

For teams using GitFlow branching:

<CardGroup cols={2}>
  <Card title="Staging Workflow" icon="flask">
    **Branch**: `develop`
    
    **Target**: Staging application
    
    **Promotion**: Automatic
    
    **Tests**: Enabled
  </Card>
  
  <Card title="Production Workflow" icon="rocket">
    **Branch**: `main` or `master`
    
    **Target**: Production application
    
    **Promotion**: Manual
    
    **Tests**: Enabled
  </Card>
</CardGroup>

### Pattern 3: Build-Once Deploy-Many

Build your application once and deploy to multiple environments:

<Steps>
  <Step title="Create workflow">
    Create a deployment workflow for your `main` branch.
  </Step>
  
  <Step title="Add staging">
    Add your staging application with automatic promotion.
  </Step>
  
  <Step title="Add production">
    Add your production application with manual promotion.
  </Step>
</Steps>

**Result**: Code merged to main is automatically deployed to staging. Once validated, promote the identical build to production with one click.

### Pattern 4: Multi-Region Deployment

Deploy to multiple regions simultaneously:

```yaml
# Configure multiple apps in the same workflow
Applications:
  - app: myapp-us-east
    rack: production-us-east
    promotion: automatic
  - app: myapp-eu-west
    rack: production-eu-west
    promotion: automatic
  - app: myapp-ap-south
    rack: production-ap-south
    promotion: automatic
```

## Integration Examples

### GitHub Actions Integration

While Convox Workflows handle most CI/CD needs, you can also integrate with GitHub Actions:

```yaml
name: Deploy to Convox

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      
      - name: Install Convox CLI
        run: |
          curl -L https://convox.com/cli/linux/convox -o /tmp/convox
          sudo mv /tmp/convox /usr/local/bin/convox
          sudo chmod +x /usr/local/bin/convox
      
      - name: Deploy to Convox
        env:
          CONVOX_HOST: console.convox.com
          CONVOX_PASSWORD: ${{ secrets.CONVOX_DEPLOY_KEY }}
          CONVOX_RACK: production
        run: |
          convox deploy -a myapp --wait
```

### GitLab CI Integration

```yaml
stages:
  - test
  - deploy

test:
  stage: test
  script:
    - npm install
    - npm test

deploy_production:
  stage: deploy
  only:
    - main
  script:
    - curl -L https://convox.com/cli/linux/convox -o /tmp/convox
    - chmod +x /tmp/convox
    - /tmp/convox deploy -a myapp --rack production --wait
  environment:
    name: production
```

## Troubleshooting

### Workflow Not Triggering

If your workflow doesn't trigger:

1. Verify the repository is connected to Convox
2. Check that the branch name matches exactly
3. Ensure webhooks are configured correctly
4. Review the workflow configuration in the Console

### Build Failures

If builds fail in workflows:

1. Check the job logs for specific error messages
2. Verify environment variables are set correctly
3. Test the build locally with `convox build`
4. Ensure all dependencies are available

### Test Failures

If tests fail:

1. Review test output in the job logs
2. Ensure test environment variables are configured
3. Check that test database and services are available
4. Verify the test command in `convox.yml` is correct
