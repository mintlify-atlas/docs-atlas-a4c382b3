The `deploy` command creates a new build from your source code and automatically promotes it to your application. This is the primary command for deploying changes to production.

## Usage

```bash
convox deploy [dir]
```

### Arguments

<ParamField path="dir" type="string" default=".">
  Directory containing the application source code (defaults to current directory)
</ParamField>

## Flags

<ParamField path="--app" type="string" default="-a">
  Application name
</ParamField>

<ParamField path="--rack" type="string" default="-r">
  Rack name
</ParamField>

<ParamField path="--description" type="string">
  Description for the deployment
</ParamField>

<ParamField path="--manifest" type="string">
  Path to manifest file (default: convox.yml)
</ParamField>

<ParamField path="--no-cache" type="boolean">
  Build without using cache
</ParamField>

<ParamField path="--id" type="boolean">
  Output only the release ID
</ParamField>

<ParamField path="--build-args" type="string">
  Build-time environment variables (can be specified multiple times)
</ParamField>

<ParamField path="--force" type="boolean">
  Force promotion even if no changes detected
</ParamField>

## Deployment Process

The deploy command performs the following steps:

1. **Build Phase**
   - Package source code
   - Upload to rack
   - Build Docker images
   - Push images to registry
   - Create new release

2. **Promotion Phase**
   - Update infrastructure (load balancers, services)
   - Roll out new containers
   - Monitor deployment status
   - Complete when all services are healthy

## Monitoring Deployments

The command streams deployment logs showing:

- Build progress
- CloudFormation stack updates
- Service status changes
- Health check results

## Rollback

If a deployment fails or causes issues, you can quickly rollback to a previous release:

```bash
# List releases to find the previous one
convox releases

# Promote the previous release
convox releases promote RPREVIOUSID
```

## Best Practices

### 1. Use Descriptive Messages

Always include a description to track what changed:

```bash
convox deploy --description "Fix payment processing bug #123"
```

### 2. Test Before Deploying

Run tests in a staging environment first:

```bash
# Deploy to staging
convox deploy --rack staging --app myapp-staging

# Run tests
convox run web --app myapp-staging npm test

# Deploy to production
convox deploy --rack production --app myapp
```

### 3. Monitor After Deployment

Watch logs after deploying:

```bash
# In another terminal
convox logs --follow
```

### 4. Use CI/CD Integration

Automate deployments in your CI/CD pipeline:

```yaml
# .github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Deploy to Convox
        run: |
          curl -L https://convox.com/cli/install.sh | sh
          convox deploy --rack production --app myapp --description "${{ github.event.head_commit.message }}"
        env:
          CONVOX_PASSWORD: ${{ secrets.CONVOX_PASSWORD }}
```

## Troubleshooting

### Build Failures

If the build fails:

1. Check build logs for errors
2. Verify Dockerfile syntax
3. Ensure all dependencies are available
4. Try `--no-cache` to eliminate cache issues

### Deployment Failures

If the promotion fails:

1. Check service health checks
2. Verify environment variables are set correctly
3. Review CloudFormation events for infrastructure issues
4. Check resource limits (CPU, memory)

### Slow Deployments

If deployments are slow:

1. Optimize Dockerfile layers for better caching
2. Use `.dockerignore` to exclude unnecessary files
3. Consider using a Docker registry closer to your rack
4. Check for large file uploads
