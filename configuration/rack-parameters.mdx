# Rack Parameters

Rack parameters are configuration settings that control various aspects of your Convox rack. These parameters allow you to customize and optimize the behavior of your applications and services at the infrastructure level.

## Overview

Rack parameters provide infrastructure-level configuration that applies to all applications running on the rack. This includes settings for:

  <Card title="Compute Resources" icon="microchip">
    Node types, capacity, scaling, and GPU configuration
  </Card>
  <Card title="Networking" icon="network-wired">

  </Card>
  <Card title="Storage" icon="hard-drive">
    EBS encryption, EFS drivers, and volume configuration
  </Card>
  <Card title="Observability" icon="chart-line">
    Logging, monitoring, and access log retention
  </Card>
</CardGroup>

## Installing Racks with Parameters

When installing a new rack through the Convox Console, you can configure parameters at installation time. This allows you to customize your rack configuration from the start.

### Using the Install Modal

During rack installation, you can set any available parameters directly in the install modal. Parameters not explicitly set will use their default values.

### Using JSON Parameter Files

You can upload a JSON file containing your desired parameter configurations:

```json
{
  "node_type": "t3.medium",
  "node_capacity_type": "on_demand",
  "build_node_enabled": "true",
  "build_node_type": "t3.large",
  "high_availability": "true",
  "node_disk": "20",
  "min_on_demand_count": "2",
  "max_on_demand_count": "10",
  "proxy_protocol": "true",
  "efs_csi_driver_enable": "true"
}
```

This example configuration:
- Uses `t3.medium` instances for regular nodes with on-demand capacity
- Enables a dedicated build node using `t3.large` instances
- Ensures high availability with a minimum of 2 on-demand nodes
- Increases node disk size to 20GB
- Enables proxy protocol for client IP preservation
- Enables EFS CSI driver for shared storage capabilities

<Note>
  You only need to include parameters you want to change from their defaults. Any parameters not specified in your JSON file will use their default values.
</Note>

## Managing Rack Parameters

### Viewing Current Parameters

To view the current rack parameters:

```bash
$ convox rack params -r rackName
access_log_retention_in_days          7
build_node_enabled                    true
build_node_min_count                  0
build_node_type                       t3.small
high_availability                     false
node_capacity_type                    on_demand
node_disk                             20
node_type                             t3.medium
```

<Note>
  This command displays the current values of all rack parameters for the specified rack.
</Note>

### Setting Parameters

To set a rack parameter:

<CodeGroup>
```bash Single Parameter
$ convox rack params set node_type=t3.large -r rackName
Setting parameters... OK
```

```bash Multiple Parameters
$ convox rack params set node_type=t3.large high_availability=true -r rackName
Setting parameters... OK
```
</CodeGroup>

<Warning>
  Some parameter changes may trigger rack updates or node replacements. Review the parameter documentation before making changes to production racks.
</Warning>

## Common Parameter Categories

### Compute Configuration

<ParamField path="node_type" type="string">
  The EC2 instance type for cluster nodes (e.g., `t3.medium`, `m5.large`)
</ParamField>

<ParamField path="node_capacity_type" type="string" default="on_demand">
  Node capacity type: `on_demand`, `spot`, or `mixed`
</ParamField>

<ParamField path="node_disk" type="number" default="20">
  Node disk size in GB
</ParamField>

<ParamField path="build_node_enabled" type="boolean" default="false">
  Enable dedicated build nodes for application builds
</ParamField>

### High Availability

<ParamField path="high_availability" type="boolean" default="false">
  Creates a cluster with redundant resources across multiple availability zones
</ParamField>

<ParamField path="min_on_demand_count" type="number">
  Minimum number of on-demand nodes when using mixed capacity type
</ParamField>

<ParamField path="max_on_demand_count" type="number">
  Maximum number of on-demand nodes when using mixed capacity type
</ParamField>

### Storage

<ParamField path="efs_csi_driver_enable" type="boolean" default="false">
  Enable the EFS CSI driver to use AWS EFS volumes
</ParamField>

<ParamField path="ebs_volume_encryption_enabled" type="boolean" default="false">
  Enable encryption for EBS volumes used by primary node disks
</ParamField>

### Networking

<ParamField path="proxy_protocol" type="boolean" default="false">
  Enable Proxy Protocol to track the original client IP address
</ParamField>

<ParamField path="private" type="boolean" default="false">
  Place nodes in private subnets behind NAT gateways
</ParamField>

<ParamField path="internal_router" type="boolean" default="false">
  Install an internal load balancer within the VPC
</ParamField>

### Observability

<ParamField path="access_log_retention_in_days" type="number" default="7">
  Retention period for Nginx access logs in CloudWatch Logs
</ParamField>

<ParamField path="fluentd_disable" type="boolean" default="false">
  Disable Fluentd installation in the rack
</ParamField>

<ParamField path="syslog" type="string">
  Endpoint to forward logs to a syslog server
</ParamField>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Parameter changes not taking effect">
    - Some parameters require rack updates to apply
    - Check rack status: `convox rack -r rackName`
    - Wait for the rack to finish updating before checking changes
    - Review CloudFormation events for any errors
  </Accordion>
  
  <Accordion title="Invalid parameter value">
    - Check parameter documentation for valid values
    - Ensure string values are properly quoted in JSON
    - Boolean values should be strings: `"true"` not `true`
    - Verify parameter names are spelled correctly
  </Accordion>
  
  <Accordion title="Cost concerns after parameter changes">
    - Review instance types and counts before applying
    - Use spot instances for non-production environments
    - Monitor AWS Cost Explorer after changes
    - Consider using mixed capacity types for cost optimization
  </Accordion>
</AccordionGroup>
